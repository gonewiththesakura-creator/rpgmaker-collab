# RPG Agents - Routing Rules

本文件定义“监督”如何将任务路由给“剧本”与“技术”。

## 总原则

1. **群里所有最终回复都由监督输出**。
2. **剧本** 处理叙事、人物、对白、任务文本。
3. **技术** 处理 RPGMaker 实现、事件、变量、地图、插件、系统逻辑。
4. 混合任务必须拆分后分别交给剧本与技术，再由监督整合。

---

## 一、直接交给剧本的任务

如果用户消息主要包含以下方向，优先路由给剧本：

- 世界观
- 人设
- 对话
- 台词
- 剧情
- 支线
- 章节结构
- 情绪氛围
- 文案润色
- 开场 / 结局 / 反转设计

### 示例
- 设计第一章开场剧情
- 帮我写 NPC 对话
- 设计主角的性格与背景
- 这个任务线怎么更有情绪张力

### 监督处理方式
- 发给剧本 agent
- 如果剧本内容明显涉及技术复杂度，再同步抄送技术 agent 评估风险

---

## 二、直接交给技术的任务

如果用户消息主要包含以下方向，优先路由给技术：

- RPGMaker 事件
- 变量 / 开关
- 插件
- UI / 菜单
- 地图事件流
- 存档 / 战斗 /任务系统
- 性能
- 实现方式
- 调试 / 修 bug

### 示例
- 这个任务系统用变量还是脚本更好
- RPGMaker 怎么做分支对话
- 帮我设计地图切换逻辑
- 这个机制实现成本高不高

### 监督处理方式
- 发给技术 agent
- 如果技术实现会影响叙事表现，再同步让剧本 agent 给体验建议

---

## 三、必须同时调剧本和技术的任务

遇到以下任务时，监督应同时调用两个 agent：

- 设计一段剧情并说明怎么落地制作
- 新系统既影响叙事又影响玩法
- 任务链既要写对白又要做事件流
- 需要决定“值不值得做”以及“怎么做”
- 需要平衡氛围、制作成本、可玩性

### 示例
- 设计主角第一次进入废墟城的完整可制作版本
- 做一个记忆碎片系统，既服务叙事也服务探索
- 第一章开场怎么写、怎么落地到 RPGMaker

### 监督处理方式
1. 给剧本 agent：让其产出叙事方案
2. 给技术 agent：让其产出实现方案与风险评估
3. 监督整合为一份统一结论

---

## 四、监督自己直接处理的任务

以下情况通常不需要下发给执行 agent，监督可以自己回答：

- 简单进度汇报
- 任务优先级安排
- 决策说明
- 路线选择
- 对外沟通
- 总结执行结果

### 示例
- 现在做到哪一步了
- 接下来先做什么更合理
- 我们为什么先做系统再做支线

---

## 五、标准工作流

### 工作流 A：剧本型任务
1. 监督收到消息
2. 判断为剧本型
3. 发送给剧本 agent
4. 收到结果后审核
5. 若有必要再让技术评估落地风险
6. 监督统一回复

### 工作流 B：技术型任务
1. 监督收到消息
2. 判断为技术型
3. 发送给技术 agent
4. 收到结果后审核
5. 若影响叙事，再让剧本补体验建议
6. 监督统一回复

### 工作流 C：混合任务
1. 监督收到消息
2. 判断为混合型
3. 同时发送给剧本和技术
4. 获取两方输出
5. 监督做冲突消解和优先级取舍
6. 输出整合方案

---

## 六、输出要求

监督整合后的最终回复，必须尽量包含：

- 当前任务理解
- 剧本建议（如果有）
- 技术建议（如果有）
- 监督的最终判断
- 下一步建议

避免把两个执行者的原文机械拼接。监督必须做真正的整合。
